# Operations Runbook

Practical runbook for reproducing, validating, and publishing the Pricing Discount Governance System.

## 1) Environment Setup

Requirements:
- Python 3.12.
- Network access only for dependency installation.
- No secrets or external data services are required for the synthetic baseline.

```bash
make install
. .venv/bin/activate
python -m pip check
```

If `make` is unavailable:

```bash
python3 -m venv .venv
. .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install -r requirements.lock
python -m pip check
```

## 2) Reproducible Runs

Default full run:

```bash
make run
```

Equivalent command:

```bash
python scripts/run_pipeline.py \
  --seed 42 \
  --customers 1200 \
  --products 28 \
  --sales-reps 45 \
  --orders 18000 \
  --start-date 2023-01-01 \
  --end-date 2025-12-31
```

Smoke run for CI/local checks:

```bash
make run-smoke
```

Reproducibility rules:
- Keep the same seed and CLI parameters to reproduce the same synthetic portfolio.
- Keep thresholds in `config/policy_thresholds.json`; do not hard-code policy values in notebooks or ad hoc queries.
- Runtime tables under `data/raw/`, `data/processed/`, and most CSV/JSON/Markdown files under `outputs/` are regenerated by the pipeline.
- The dashboard, report PDF, and chart pack are the reviewable release artifacts.

## 3) Pipeline Execution Order

`scripts/run_pipeline.py` runs these stages:

1. Generate synthetic raw tables.
2. Save raw CSVs to `data/raw/`.
3. Validate raw schema, PK/FK integrity, date logic, and pricing bounds.
4. Build DuckDB warehouse models and SQL marts.
5. Build pandas enriched and pricing feature tables.
6. Build customer risk scoring outputs.
7. Validate processed tables and metric contracts.
8. Run profiling, formal analysis, visualization pack, and build the dashboard candidate.
9. Run final validation review.
10. Evaluate the release gate, publish the GitHub Pages copy only on success, and write `outputs/run_manifest.json`.

The pipeline stops on failed raw, SQL, processed, metric-contract, or release-gate checks. Start debugging from the file named in the exception.

## 4) Data Contract

Raw source contract:

| Table | Required key | Required relationships |
|---|---|---|
| `customers` | `customer_id` | referenced by `orders.customer_id` |
| `products` | `product_id` | referenced by `order_items.product_id` |
| `sales_reps` | `sales_rep_id` | referenced by `orders.sales_rep_id` |
| `orders` | `order_id` | references `customers`, `sales_reps` |
| `order_items` | `order_item_id` | references `orders`, `products` |

Operational rules:
- `order_items` is the canonical transaction grain.
- Joins must preserve order-item row counts unless the target table is explicitly aggregated.
- Percentage/share metrics must stay in `[0, 1]`.
- Monetary metrics use modeled USD for presentation and should be reconciled from `line_revenue`.
- Customer scoring covers transacting customers only.
- Non-transacting customers may exist in `customers` but must not enter behavior/risk metrics.

See [data_dictionary.md](data_dictionary.md) and [data_model_and_grain.md](data_model_and_grain.md) for full definitions.

## 5) Testing and Quality Gates

Run before changing release artifacts:

```bash
make check   # lint, format-check, typecheck, compile, tests (90% gate), preflight
make audit   # dependency vulnerability scan (network required)
```

Or run the stages individually: `make lint`, `make format-check`, `make typecheck`,
`make test`, `make preflight`.

Run a release gate check after `make run`:

```bash
python scripts/release_gate.py
```

Primary diagnostics:

| File | Use |
|---|---|
| `outputs/raw_validation_report.csv` | raw schema, PK/FK, date, and pricing checks |
| `outputs/warehouse/sql_validation_report.csv` | SQL mart grain and reconciliation checks |
| `outputs/processed_validation_report.csv` | processed table bounds, grain, and population checks |
| `outputs/metric_contract_validation.csv` | published metric contract checks |
| `outputs/final_validation_summary.json` | final readiness state and blocker counts |
| `outputs/release/release_gate_report.json` | release policy pass/fail reasons |
| `outputs/run_manifest.json` | run configuration, row counts, and validation flags |

## 6) Failure Triage

Use this order:

1. Raw validation failure: inspect missing columns, duplicate keys, FK gaps, invalid dates, or price arithmetic in `outputs/raw_validation_report.csv`.
2. SQL warehouse failure: inspect no-silent-drop checks and subtotal reconciliation in `outputs/warehouse/sql_validation_report.csv`.
3. Processed validation failure: inspect grain uniqueness, score/share bounds, weighted margin, and customer population alignment in `outputs/processed_validation_report.csv`.
4. Metric contract failure: compare the failing metric in `outputs/metric_contract_validation.csv` with `config/metric_contracts.json`.
5. Release gate failure: inspect `outputs/release/release_gate_report.md` for the blocking policy rule.
6. Preflight failure: fix missing required artifacts, stale dashboard copy, broken Markdown links, or oversized/forbidden tracked files.

## 7) Security and Privacy

- The checked-in baseline is synthetic and contains no real customer, product, sales rep, or transaction data.
- Do not commit real commercial data, customer identifiers, credentials, API keys, or private contracts.
- For real-data adaptation, replace identifiers with stable pseudonymous keys before writing local CSVs.
- Treat generated dashboards and PDFs as decision artifacts: review outputs for sensitive labels before publishing.
- `.venv/`, raw/processed runtime data, validation CSVs, JSON summaries, and most runtime Markdown outputs are ignored by git.

## 8) Release Process

Release candidate sequence:

```bash
make install
make run
make report
make audit
make check
python scripts/release_gate.py
```

Publish only when:
- `release_gate_report.json` has `"gate_passed": true`.
- `final_validation_summary.json` has no failed blocker checks.
- Dashboard parity check passes: `docs/pricing-discipline-command-center.html` matches `outputs/dashboard/pricing-discipline-command-center.html`.
- Dashboard SHA-256 matches the artifact recorded by the final review and release gate.
- Chart pack and PDF have been rebuilt after the final pipeline run.
- The synthetic-data caveat remains visible in report/dashboard documentation.

Reviewable release artifacts:
- `docs/index.html`
- `docs/pricing-discipline-command-center.html`
- `docs/vendor/chart.umd.min.js`
- `outputs/dashboard/pricing-discipline-command-center.html`
- `outputs/dashboard/vendor/chart.umd.min.js`
- `outputs/graphs/*.png`
- `outputs/reports/pricing_discount_governance_report.pdf`

## 9) Maintenance Notes

- Update `config/policy_thresholds.json` and docs together when policy thresholds change.
- Update `config/metric_contracts.json` when adding/removing published metrics.
- Add or update SQL query examples in [sql_query_patterns.md](sql_query_patterns.md) when a mart becomes a recurring stakeholder query.
- Keep documentation concise; operational details belong here, while metric definitions belong in the data dictionary.
